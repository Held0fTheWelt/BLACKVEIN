Warum diese Algorithmen wichtig sind

Die Player-UI wirkt wie ein einziger Flow. Intern laufen aber mehrere Verträge zusammen: Auth im Backend, Save-Slot-Bindung in der Datenbank, Play-Service-Run, World-Engine-Story-Session, Observability und Rückgabe eines verständlichen Bundles an den Browser.

Merksatz: Der Session-Algorithmus baut die Verbindung auf. Der Turn-Algorithmus verarbeitet eine einzelne Eingabe und darf erst dann committen, wenn der autoritative Engine-Pfad sie akzeptiert oder bewusst als recoverable Ergebnis behandelt.

Session-Algorithmus: Start oder Resume

  1. User validieren.JWT prüfen, Account laden, Ban-/Aktivstatus beachten. Ohne gültigen User entsteht keine Player-Session.
  2. Run-ID bestimmen.Entweder kommt run_id aus dem Request, oder das Backend erstellt über den Play-Service einen neuen Run aus template_id oder runtime_profile_id.
  3. Save Slot suchen.Für user_id + run_id wird ein deterministischer Slot gesucht. Enthält er eine gültige runtime_session_id, versucht das Backend Resume.
  4. Bei Resume: Engine-State holen.Der World-Engine-State entscheidet, ob die Session wirklich noch existiert. Fehlt sie dort, meldet das Backend session_lost statt still neu zu erfinden.
  5. Bei Neustart: Run-Details und Runtime-Handoff prüfen.Template, Runtime-Profil, ausgewählte Spielerrolle und Content-Provenance werden geprüft, bevor ein Story-Modul kompiliert wird.
  6. Runtime Projection bauen.Das Backend übersetzt das Template in eine Engine-verträgliche Projektion: Modul, Rollenbindung, Sprache, Content-Provenance.
  7. World-Engine Story Session erstellen.POST /api/story/sessions erzeugt die autoritative Session und den Opening-Turn. Fehlt canonical opening evidence, schlägt der Start hart fehl.
  8. Binding speichern und Bundle zurückgeben.game_save_slots speichert run_id, template_id, module_id und runtime_session_id. Danach fragt das Backend den State ab und liefert das Player-Bundle.

Der entscheidende Kontrollpunkt liegt beim Opening-Turn: Ohne kanonische Opening-Evidence wird keine scheinbar gesunde Session zurückgegeben.

Turn-Algorithmus: eine Spielereingabe

  1. Eingabe normalisieren.Das Backend akzeptiert player_input oder input, trimmt Leerraum und lehnt leere Eingaben ab.
  2. Session sicherstellen.Vor jedem Turn läuft der Resume-Teil des Session-Algorithmus: Save Slot lesen, runtime_session_id finden, Engine-State prüfen.
  3. Trace vorbereiten.Wenn Langfuse verfügbar ist, startet das Backend backend.turn.execute und hängt Hash, run_id, session_id und lokale/Production-/Staging-Metadaten an.
  4. Engine-Turn ausführen.Das Backend ruft POST /api/story/sessions/{session_id}/turns im World-Engine-Dienst auf. Container nutzen die interne Service-URL.
  5. World-Engine verarbeitet autoritativ.Die Engine hasht die Eingabe, klassifiziert den Trace, startet den Turn-Span und führt StoryRuntimeManager.execute_turn aus.
  6. Lifecycle durchlaufen.Received → interpreted → generated/resolved → validated → committed → projected → persisted → observed.
  7. State und Turn zurückgeben.Das Backend holt nach dem Engine-Turn den aktuellen State, aktualisiert Observability und liefert das Bundle an die Player-UI.

Lifecycle in drei verständlichen Phasen

1. Vor dem Commit

Input wird interpretiert, Kontext/RAG werden gesammelt, die Engine erzeugt oder resolved ein mögliches Ergebnis. Noch ist nichts kanonisch.

2. Commit-Entscheidung

Validatoren und Runtime-Policy entscheiden, ob ein Ergebnis committed, recoverable abgelehnt oder als Fehler behandelt wird.

3. Nach dem Commit

State, History, sichtbare UI-Blöcke, Diagnostics und Observability werden aktualisiert. Scores erklären nur, sie mutieren keine Wahrheit.

Fehler- und Fallback-Lesart

  • Leerer Input: Backend lehnt vor der Engine ab.
  • Session verloren: Save Slot zeigt auf eine Story Session, die World-Engine nicht mehr kennt; Restart erforderlich.
  • Governance blockiert: Engine meldet Governance- oder Runtime-Konfigurationsfehler; keine falsche Erfolgshülle bauen.
  • Recoverable Turn: Die Engine kann ein spielersichtbares, aber als recoverable markiertes Ergebnis persistieren.
  • Observability-Ausfall: Runtime läuft weiter; lokale Traces oder fehlende Scores verändern Commit/Readiness nicht.

Direkt zu den Relationen

  • Datenmodell & Relationen — welche Tabellen diesen Ablauf verbinden.
  • Runtime-Pfad — welche Komponente in welchem Schritt autoritativ ist.
  • Observability & Evidence — warum Scores keine Runtime-Wahrheit mutieren.
