# W2.3 Regression Review: Memory and Context Layer

**Review Date**: 2026-03-28
**Reviewer**: Claude Code
**Review Principle**: Based on actual runtime wiring, not implementation intent

---

## Executive Summary

**W2.3 Overall Status**: **PARTIAL** ✋

W2.3 establishes well-designed, bounded, deterministic models for a memory and context layer. However, **none of these layers are integrated into the actual SessionState or runtime flow**. They exist as isolated pure functions suitable for future integration but provide no practical coherence value to longer sessions TODAY.

**Gate Decision**: **NOT READY** 🛑

W2.3 is not ready to pass the gate because:
1. No W2.3 layers are part of SessionState
2. No W2.3 derivation functions are called anywhere in the runtime
3. SessionHistory is never instantiated or accumulated during real gameplay
4. ShortTermTurnContext is never derived from real turn results
5. The layers exist as "good foundation" but not as "practical runtime reality"

**Missing**:

2. SessionState field or session-level storage

**Strongest Evidence of Problem**:

## Overall Result

### Overall W2.3 Status
**PARTIAL** — All five layers are well-designed, bounded, deterministic, and test-covered. But **none are integrated into the actual SessionState or runtime flow**. They exist as pure functions suitable for future integration but provide no practical value to longer sessions today.

### Gate Decision
**NOT READY** 🛑

W2.3 does not yet meet the gate criteria. The core requirement is:
> "longer sessions remain more coherent, bounded, and reusable in runtime reality rather than only as helper models"

W2.3 provides helper models but zero runtime reality. The layers are not reused in any actual session.

### Top Blockers

1. **No SessionState Integration** (Critical)
   - SessionState does not include fields for any W2.3 layers
   - This is the foundational blocker for all five subsections
   - Must add: history, short_term_context, progression, relationships, lore_context to SessionState (or use a wrapper)

2. **No Derivation Points** (Critical)
   - No code calls any W2.3 derivation function
   - turn_executor.py never calls build_short_term_context()
   - No one ever calls derive_progression_summary(), derive_relationship_axis_context(), or derive_lore_direction_context()
   - Must add: actual call sites in the runtime flow

3. **No Accumulation Path** (Critical)
   - SessionHistory is never instantiated or maintained
   - add_entry() is never called
   - Without SessionHistory, all downstream layers (progression, relationships, lore) have no input data

4. **No Practical Use** (Critical)
   - Even if layers were derived, there's no evidence they're used for anything
   - No decision-making consults relationship escalation
   - No narrative shaping uses lore context
   - No scene selection uses progression state

### False-Completeness Risks

- ✋ **"We have all five layers, so W2.3 is complete"** — FALSE. Having models defined is not completion.
- ✋ **"All tests pass, so W2.3 works"** — FALSE. Tests verify isolated correctness, not integration.
- ✋ **"The design is sound"** — TRUE, but design alone is not implementation in runtime reality.
- ✋ **"This is a good foundation"** — TRUE, but foundations aren't released as features. They must be integrated.

### Minimal Repair Sequence

To make W2.3 ready for gate:

1. **Phase 1: Wiring (Prerequisite for all other work)**
   - Add fields to SessionState for: history (SessionHistory), short_term_context (optional), progression, relationships, lore_context
   - OR create SessionContextLayers wrapper struct if SessionState is too large
   - Initialize SessionHistory when session starts
   - Initialize other layers (progression, relationships, lore_context) as empty/None
   - Commit: "refactor(w2): integrate W2.3 layers into SessionState"

2. **Phase 2: Accumulation (Enables W2.3.1-W2.3.2)**
   - Add call to build_short_term_context() after turn completes in turn_executor.py
   - Add call to SessionHistory.add_from_short_term_context() after ShortTermTurnContext is built
   - Verify SessionHistory accumulates correctly across turns
   - Commit: "feat(w2): wire session history accumulation into turn execution"

3. **Phase 3: Derivation (Enables W2.3.3-W2.3.5)**
   - Add logic to derive ProgressionSummary from SessionHistory at appropriate intervals (after each turn? every N turns?)
   - Add logic to derive RelationshipAxisContext from SessionHistory
   - Add logic to derive LoreDirectionContext from module + progression + relationships
   - Update SessionState fields with derived contexts
   - Commit: "feat(w2): wire context derivation into session state updates"

4. **Phase 4: Verification**
   - Run full session with real turn execution
   - Verify all W2.3 layers accumulate and derive
   - Verify layers remain bounded (history max 100, relationships max 10, lore_context max 15)
   - Verify longer sessions remain coherent (new turns see accumulated history, not raw replays)
   - Add integration test showing full W2.3 flow in action
   - Commit: "test(w2): verify W2.3 integration with full session flow"

## Recommendation

**W2.3 is well-founded but not ship-ready. The gate remains CLOSED.**

Proceed to repair sequence only after explicitly deciding:
1. How SessionState should accommodate W2.3 layers (field list vs wrapper struct)
2. When/how derivation should be triggered (after each turn? periodic?)
3. Where relationship/lore context will be used (AI request assembly? narrative decisions?)

Do not merge W2.3 without clear integration wiring. Isolated models are not features.
