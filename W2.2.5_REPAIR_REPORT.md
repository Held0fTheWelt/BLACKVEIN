## Asymmetries Fixed

### Issue 1: Event Log Asymmetry (FIXED ✓)
**Original Problem:**
- `turn_completed` event included `"guard_outcome"` in payload
- `turn_failed` event did NOT include `"guard_outcome"` in payload
- Event consumers had to infer error-path guard outcome or default it

**Fix Applied:**
- Added `"guard_outcome": GuardOutcome.STRUCTURALLY_INVALID.value` to `turn_failed` event payload (turn_executor.py:595–603)
- Now both success and error paths emit canonical guard outcome in event payload
- Cross-surface consistency: event logs now carry same guard outcome as TurnExecutionResult

### Issue 2: Error-Path Test Coverage (FIXED ✓)
**Original Problem:**
- No tests verified error-path guard outcome consistency
- AIDecisionLog error-path semantics were not tested for cross-surface alignment
- Implicit assumptions about guard outcome in error cases

**Fix Applied:**
- Added `test_error_path_result_has_structurally_invalid_guard_outcome` — verifies result.guard_outcome
- Added `test_error_path_aidecisionlog_has_error_validation_outcome` — verifies log.validation_outcome=ERROR
- Added `test_error_path_cross_surface_consistency` — comprehensive cross-surface verification
- All 3 tests verify that failed turns carry explicit, consistent guard outcome semantics

### Issue 3: Guard_Notes Asymmetry (REVIEWED, INTENTIONAL ✓)
**Original Asymmetry:**
- Success path: `guard_notes = "{count} error(s); {outcome_label}: {sample}"` (outcome_label = guard_outcome.value)
- Error path: `guard_notes = "{count} error(s); {error_type}: {sample}"` (error_type = adapter_error, parse_error, etc.)

**Analysis:**
- This asymmetry is **semantically intentional**, not accidental
- Success path tracks validation errors + their guard outcome classification
- Error path tracks system errors + their error type classification
- Both follow the same normalization pattern: "N error(s); TYPE: details"
- Distinction is meaningful: validation outcome vs error type are different concerns

**Decision:** RETAINED INTENTIONALLY
- The asymmetry is semantically correct
- Tests confirm the pattern (test_guard_notes_format_normalized_with_error_count_and_label)
- Future modifications should preserve this distinction

## Canonical Failed-Turn Guard Outcome

**Failed Turn Guard Outcome Value:** `GuardOutcome.STRUCTURALLY_INVALID` (string value: `"structurally_invalid"`)

**When Used:**
- All error-path turns (parse failure, adapter error, system error)
- Empty decision proposals (no deltas proposed)
- Any execution_status != "success"

**Mapping to Validation Semantics:**
- `GuardOutcome.STRUCTURALLY_INVALID` → `AIValidationOutcome.ERROR` in AIDecisionLog
- This mapping is canonical and enforced in `_create_decision_log()` and `_create_error_decision_log()`

**Rationale:**
- STRUCTURALLY_INVALID is the correct classification for error-path turns
- Distinct from REJECTED (which is for validation failure in success path)
- Explicit and canonical across all runtime surfaces
