# ADR-0031: Environment Configuration Governance (.env)

## Status

Accepted

## Date

2026-05-04

## Intellectual property rights

Repository authorship and licensing: see project LICENSE; contact maintainers for clarification.

## Privacy and confidentiality

This ADR contains guidelines for secrets management. Implementers must follow repository confidentiality policies and avoid committing live credentials to .env examples or documentation.

## Context

The `.env` file is the single source of truth for runtime configuration across Backend, Frontend, Administration Tool, and World Engine services. Without clear governance:

1. **Inconsistency risks**: Services use different initialization patterns (environment variables, database, env files), causing configuration drift
2. **Hidden failures**: Missing or misconfigured variables cause silent failures (e.g., Langfuse disabled instead of erroring)
3. **Long-term maintainability**: New developers don't know which variables are required, optional, or service-specific
4. **Integration quality**: Features like Langfuse tracing fail silently when env variables are missing (real incident: LANGFUSE_ENABLED omitted)

**Incident example:**
- Langfuse integration was complete in code but non-functional
- Root cause: `LANGFUSE_ENABLED` not in `.env`
- World Engine adapter skipped initialization silently
- Discovery took 1+ hour of debugging
- _Could have been prevented by clear governance rules_

## Decision

### 1. Classification of Variables

All `.env` variables MUST fall into one of these categories:

| Category | Required? | Source | Example | Rule |
|----------|-----------|--------|---------|------|
| **Platform Secrets** | YES | Generated by `python docker-up.py init-env` | `SECRET_KEY`, `JWT_SECRET_KEY` | Auto-generated once, preserved across resets. Never manually edit. |
| **Service Enables/Disables** | YES | Manual (see table below) | `LANGFUSE_ENABLED=true` | Must be explicitly set; defaults to `false` or validation errors. NO silent disabling. |
| **Provider Credentials** | NO (optional per provider) | Manual or API | `OPENAI_API_KEY`, `ANTHROPIC_API_KEY` | Leave blank unless that provider is used. |
| **Provider URLs** | NO (defaults provided) | Manual (override defaults) | `OPENAI_BASE_URL`, `OLLAMA_BASE_URL` | Pre-filled with public defaults. Override only for self-hosted/proxies. |
| **Network/Deployment** | YES (varies by context) | Manual | `CORS_ORIGINS`, `FRONTEND_URL`, `BACKEND_RUNTIME_CONFIG_URL` | Docker Compose sets sensible defaults; override for non-standard deployments. |
| **Feature Configuration** | NO (sensible defaults) | Manual override | `MAIL_ENABLED`, `REGISTRATION_REQUIRE_EMAIL` | Optional. Defaults favor development (mail off, no email required). |

### 2. Service Enable/Disable Variables

These variables **MUST** be explicitly set in `.env` for each service that requires initialization:

| Variable | Service | Default | Rule | Consequence if Missing |
|----------|---------|---------|------|------------------------|
| `LANGFUSE_ENABLED` | Backend + World Engine | `false` (implicit, not silent) | **MUST be `true` or `false` explicitly** | ❌ Langfuse adapter skips init; traces never created |
| `MAIL_ENABLED` | Backend | `false` (sensible default) | Set to `1` or `true` for production | ❌ Password reset emails don't send |
| `REQUIRE_EMAIL_VERIFICATION_FOR_LOGIN` | Backend | `true` (security default) | Explicitly set for clarity | ⚠️ Security context unclear; prefer explicit |
| `DEV_SECRETS_OK` | Backend (local dev only) | `false` | Set to `1` for local dev/test | ❌ Dev seed commands refuse to run |
| `RUN_STORE_BACKEND` | World Engine | `json` (sensible default) | Explicit for clarity | ⚠️ Silent fallback; prefer explicit |

### 3. Initialization Sequence

Every service **MUST** check for service-enable variables at startup and fail loudly if misconfigured:

```python
# Backend example (factory_app.py):
import os
env_enabled = os.getenv("LANGFUSE_ENABLED", "").lower() in ("true", "1")
if os.getenv("LANGFUSE_SECRET_KEY"):
    if not env_enabled:
        raise RuntimeError(
            "LANGFUSE_SECRET_KEY set but LANGFUSE_ENABLED!=true. "
            "Set LANGFUSE_ENABLED=true in .env or remove the secret."
        )
```

**Rule:** If a variable has a value that implies a service should be enabled (e.g., `LANGFUSE_SECRET_KEY` exists), but the enable flag is absent or `false`, **raise an error, not a warning**.

### 4. What goes in .env (Summary)

**MUST add to .env:**
- Platform secrets (auto-generated by `docker-up.py init-env`)
- Service enable/disable flags (explicit `true`/`false`)
- Provider API keys (if using that provider)
- Network URLs (if non-standard deployment)
- INTERNAL_RUNTIME_CONFIG_TOKEN (auto-generated; shared between Backend and World Engine)

**MUST NOT commit to .env (but safe to add locally):**
- Real OpenAI, Anthropic, etc. API keys (use `.env.example` or documentation for keys)
- Database URLs with passwords (use defaults or use env var only in CI/prod)

**Optional in .env:**
- Feature config flags (mail, verification, etc.) — safe to leave at defaults
- Provider URLs (use public defaults unless self-hosting)

### 5. Adding New .env Variables

When adding a new feature that requires configuration:

1. **Classify the variable** (see table above)
2. **Add documentation in .env** with inline comments explaining purpose and default
3. **Add initialization check** in the owning service (e.g., Backend or World Engine)
4. **If it controls a feature**, add a service enable/disable variable with explicit `true`/`false`
5. **Update this ADR** with the new variable in the table
6. **Add a test** that verifies initialization fails if the variable is misconfigured

### 6. Validation Rules

Services **MUST** validate `.env` on startup:

```
Backend startup:
  ✓ FLASK_APP set
  ✓ JWT_SECRET_KEY non-empty
  ✓ If LANGFUSE_SECRET_KEY set: LANGFUSE_ENABLED=true
  ✓ If LANGFUSE_ENABLED=true: LANGFUSE_SECRET_KEY non-empty
  → Error if validation fails; don't start

World Engine startup:
  ✓ BACKEND_RUNTIME_CONFIG_URL reachable
  ✓ INTERNAL_RUNTIME_CONFIG_TOKEN non-empty
  ✓ If LANGFUSE_ENABLED=true: credentials fetched from Backend
  → Error if validation fails; don't start
```

### 7. Breaking Silences (High Priority)

These patterns **MUST raise errors, not warnings**:

- Service enable flag is absent when a credential exists
- Service tries to initialize but required config is missing
- .env variable has a placeholder value (e.g., `your-key-here`)
- Backend healthcheck passes but critical .env variable is misconfigured

## Consequences

**Positive**
- ✓ New developers see exactly which variables are required
- ✓ Feature implementation includes .env governance rules from day one
- ✓ Silent failures become loud, actionable errors
- ✓ Long-term consistency: all services follow the same pattern
- ✓ Reduced onboarding time: "check this ADR for new variables"

**Negative / Risks**
- ⚠ More verbose .env file (tradeoff for clarity)
- ⚠ Services that ignore these rules will regress (requires code review discipline)

**Follow-ups**
- Create `.env.example` documenting all variables and defaults
- Add linting script to validate .env syntax and required keys
- Add GitHub Actions check: `.env` governance validation in CI

## Testing

### Verification Checklist

**New .env variable added:**
- [ ] Documented in `.env` with inline comment (purpose, default, requirement)
- [ ] Classified in this ADR (table above)
- [ ] Service checks value on startup (no silent skip or default)
- [ ] If service enable flag: test that missing flag raises error
- [ ] If credential: test that credential without enable flag raises error

**Breaking silence test:**
- [ ] Backend + World Engine start cleanly with valid .env
- [ ] Backend fails loudly if LANGFUSE_SECRET_KEY set but LANGFUSE_ENABLED!=true
- [ ] World Engine fails loudly if LANGFUSE_ENABLED=true but Backend unreachable
- [ ] Both services explicitly log which variables were loaded/validated

**Regression test:**
- [ ] Run `python docker-up.py up` with fresh .env → all services healthy
- [ ] All services report their .env configuration on startup
- [ ] No silent failures; all errors are explicit
