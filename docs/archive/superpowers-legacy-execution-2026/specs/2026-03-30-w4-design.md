# W4 Design: MVP Hardening, Quality, First Real Vertical Slice

> **Status:** Design approved by user. Ready for spec review and implementation planning.

## Executive Summary

W4 transforms the technically working prototype into a hardened, presentable MVP. Five critical gates form the operational readiness tier and must close in sequence before W4 is complete. Two quality enhancement packages (Tier 2) are important but do not block first W4 closure. This spec defines a **Strict Sequential Gate Model** for cost-efficient implementation using focused agents.

## W4 Completion Rule

**W4 is complete only when:**
- The MVP is stable end-to-end (System Tests gate)
- Sessions persist durably across interruption (Persistence gate)
- UI is clear and usable (Usability gate)
- Presentation is repeatable and confident (Demo gate)
- Scope is disciplined (Boundary gate)
- Balancing and AI quality are optimized within MVP constraints (quality tier, parallel/deferred)

## Tier Structure

### Tier 1: Operational Readiness (Sequential Gates)
Five gates must close in strict order. Each gate must pass before next begins.

1. **System Tests / End-to-End Tests** — Establish vertical slice confidence
2. **Session Persistence Hardening** — Enable durable save/load/resume
3. **UI Usability Improvement** — Clarify operator experience
4. **Demo / Presentation Script** — Define reproducible showcase paths
5. **MVP Boundary Review** — Lock scope and reject drift

### Tier 2: Quality Enhancement (Parallel/Deferred)
Run in parallel with Tier 1 or after gates close; do not block Tier 1 completion.

6. **Balancing / Fine-Tuning** — Escalation, coalition shifts, character consistency
7. **AI Quality Improvement** — Prompt structure, context selection, validation, routing

## Gate Definitions

### Gate 1: System Tests / End-to-End Tests

**Problem:** Strong focused tests exist, but confidence gap in full session lifecycle behavior, especially longer runs and failure paths.

**Scope:**
- Write comprehensive E2E test suite covering:
  - Session start → multi-turn progression (5–10 turns)
  - Typical escalation paths (pressure build, coalition shifts)
  - Error paths (invalid input, AI failures, guard rejections)
  - Recovery behavior (degraded mode, fallback decisions)
  - Session ending (natural conclusion, forced termination)
- Verify no hidden breakage in longer runs
- Establish confidence baseline for subsequent gates

**Success Criteria:**
- Comprehensive E2E test suite covers all critical scenario types: multi-turn progression, escalation paths, error paths, recovery behavior, and session termination
- Multi-turn progression tested (5–10 turn sessions)
- Error and recovery paths exercised (invalid input, AI failures, guard rejections, degraded mode)
- Manual smoke test: Multi-turn sessions complete with expected story progression and no crashes
- All existing tests still pass (no regressions)

**Deliverable:**
- `tests/test_e2e_god_of_carnage_full_lifecycle.py` — comprehensive E2E suite
- Coverage report showing test coverage by scenario type

**Gate Closure Conditions:**
1. All E2E tests pass
2. Manual smoke test of 10-turn session succeeds with predictable output
3. No regression in existing test suite
4. Sign-off: "System is stable end-to-end"

### Gate 2: Session Persistence Hardening

**Problem:** Currently in-memory only. No durable save/load/resume. Cannot recover from interruption or inspect past sessions reproducibly.

**Scope:**
- Implement durable persistence layer:
  - Session serialization (to disk/DB)
  - Session deserialization with full context recovery
  - Save triggers (explicit save, auto-save on turn completion)
  - Resume with full state, metadata, diagnostics
- Preserve:
  - `SessionState` (module_id, scene_id, turn_counter, canonical_state, context_layers)
  - Full diagnostic metadata (ai_decision_logs, turn execution results, degradation markers)
  - Runtime context for reproducible inspection
- Verify E2E tests still pass after persistence integration

**Success Criteria:**
- Stop session after turn N, serialize to disk
- Deserialize and resume: all state identical
- Resume session: can execute turn N+1 with full context
- Diagnostics remain accessible and accurate
- All E2E tests still pass

**Deliverable:**
- `app/runtime/session_persistence.py` — persistence layer (serialize/deserialize)
- `app/services/persistence_service.py` — service layer (save/load/resume orchestration)
- Integration tests: 10+ persistence tests covering save/load/resume scenarios
- Documentation: persistence API and restore workflow

**Gate Closure Conditions:**
1. Persistence tests pass (serialize/deserialize/resume)
2. E2E tests pass after persistence integration
3. Manual test: stop mid-session, resume, continue without data loss
4. Sign-off: "Sessions persist durably across interruption"

### Gate 3: UI Usability Improvement

**Problem:** Shell is functional but lacks clarity. Panels visible but operator flow, changes, and diagnostic priority are not obvious. UI feels "developer-shaped."

**Scope:**
- Clarify operator flow:
  - What is the primary action loop? (read → decide → execute → observe result)
  - What changed this turn? (highlight recent changes, state deltas)
  - Which diagnostics matter most? (priority signal on debug panel)
- Reduce cognitive load:
  - Simplify panel layout; hide less-critical detail by default
  - Add section headers and guidance text
  - Improve visual hierarchy (what's story? what's diagnostics? what's control?)
- Verify usability without expert guidance

**Success Criteria:**
- Non-developer can follow story flow without confusion
- Can identify what changed after each turn
- Diagnostic panel shows priorities clearly
- Session shell loads without developer explaining the layout
- All tests pass

**Deliverable:**
- Updated `app/web/templates/session_shell.html` — clarified layout and visual hierarchy
- `docs/UI_USABILITY.md` — operator flow guide and panel priorities
- Screenshots showing before/after clarity improvements

**Gate Closure Conditions:**
1. UI walkthrough: non-developer follows story flow without guidance
2. Can answer "what changed?" and "why does it matter?" without dev explanation
3. All tests pass
4. Sign-off: "UI is clear and usable"

---

### Gate 4: Demo / Presentation Script

**Problem:** Runs work ad-hoc but lack deliberate shape. No reproducible showcase paths, no defined fallback moments, no predictable explanation points. Demo readiness is uncertain.

**Scope:**
- Define 3 reproducible demo paths, each with:
  - Step-by-step operator input script
  - Expected outputs at key checkpoints
  - Fallback explanation if unexpected behavior occurs
  - Timing estimates and key moments to highlight

**Three Demo Paths:**
1. **Good Run:** Stable 5–7 turn scenario showing coherent story progression, character consistency, readable escalation
2. **Stressed Run:** 8–12 turn scenario with pressure build, coalition shifts, degradation handling, recovery moments
3. **Failure/Recovery Demo:** Deliberate error (invalid input, AI failure, guard rejection) with graceful recovery and diagnostic visibility

**Success Criteria:**
- Each path has documented operator script
- Each path executes predictably (input → expected output)
- Operator can explain what's happening at each moment
- Fallback explanations address likely unexpected behaviors
- All paths complete without crashes

**Deliverable:**
- `docs/DEMO_SCRIPTS.md` — 3 complete demo scripts with inputs, expected outputs, timing, explanation points
- Session logs/recordings for each demo path (proof of reproducibility)
- Fallback reference guide (common issues and recovery explanations)

**Gate Closure Conditions:**
1. All 3 demo scripts execute reproducibly with stable checkpoints at expected moments
2. Operator can confidently run any demo without external help and explain what's happening
3. Timing variations and content variations within acceptable bounds for an AI-driven MVP (exact iteration may vary, but progression and key moments remain stable)
4. Sign-off: "Presentation is repeatable and confident"

---

### Gate 5: MVP Boundary Review

**Problem:** Scope boundary is clear on paper but risks gradual drift through "small useful additions" that are not essential for MVP proof.

**Scope:**
- Audit codebase for scope creep:
  - Identify any features/code not essential to: stability, traceability, persistence, usability, demo readiness
  - Classify: keep (MVP-critical) vs. defer (nice-to-have, W4+ or later)
- Document MVP boundary explicitly:
  - What's included (and why)
  - What's deferred (and when it will be done)
- Lock scope: reject new additions unless they improve one of the 5 readiness dimensions

**Success Criteria:**
- Scope document identifies all major code sections
- Boundary decision rationale documented for each section
- No MVP-critical features deferred
- Deferred features have clear W4+ scheduling
- Codebase has no "creeping" features beyond boundary
- All tests pass

**Deliverable:**
- `docs/MVP_BOUNDARY.md` — scope audit, included features, deferred features with W4+ timing
- `docs/NEXT_CONTENT_WAVE.md` — definition of "ready for next content wave" based on completed gates

**Gate Closure Conditions:**
1. Scope audit complete and documented
2. Boundary decisions reviewed and approved
3. No regressions in test suite
4. Sign-off: "Scope is disciplined; ready for next content wave"

## Implementation Strategy

### Cost Optimization (Approach A: Strict Sequential)

**Why Strict Sequential?**
- **Single-threaded focus:** Agents work one gate at a time, no parallel coordination overhead
- **Cheap models:** Each gate can use lower-cost/faster models for mechanical work (test writing, persistence plumbing)
- **Clear scope:** No context switching, no "wait for upstream" overhead
- **Linear progression:** Each gate closure triggers next gate start; no idle time

**Agent Workflow:**
1. Agent A: Gate 1 (System Tests) — test writing, test execution, verification
2. Agent B: Gate 2 (Persistence) — persistence layer implementation, integration
3. Agent C: Gate 3 (UI Usability) — UI clarification, walkthrough verification
4. Agent D: Gate 4 (Demo/Script) — demo path definition, reproducibility testing
5. Agent E: Gate 5 (Boundary Review) — scope audit, documentation

**Tier 2 Work (Balancing/AI Quality):**
- Can start in parallel with Gate 1 or after any gate closure (independent work)
- Does not block Tier 1 completion
- Optimized within MVP constraints

### Success Metrics

| Gate | Metric | Target |
|------|--------|--------|
| 1    | E2E scenario coverage | All 5 scenario types (progression, escalation, error, recovery, termination) tested and passing |
| 2    | Persistence restore | 100% state fidelity, zero data loss |
| 3    | UI clarity | Non-developer can follow without guidance |
| 4    | Demo reproducibility | 3 paths with stable checkpoints, operator-confident narration, acceptable variation in timing/content |
| 5    | Scope discipline | Zero drift features, clear deferral list |

### Timeline Estimate

| Gate | Effort | Dependencies |
|------|--------|--------------|
| 1    | 2–3 sprints | None (builds on W3) |
| 2    | 2–3 sprints | Gate 1 complete |
| 3    | 1–2 sprints | Gate 2 complete |
| 4    | 1–2 sprints | Gate 3 complete |
| 5    | 1 sprint | All gates in progress/complete |

**Total: 7–11 sprints** (strictly sequential, no parallelization overhead)
