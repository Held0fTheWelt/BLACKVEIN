# At-rest encryption evidence and completion plan

Status date: 2026-05-17

Normative decision records:

- [ADR-0047: At-rest encryption evidence boundary](../ADR/adr-0047-at-rest-encryption-evidence-boundary.md)
- [ADR-0051: Storage-layer encryption governance](../ADR/adr-0051-storage-layer-encryption-governance.md)

## Conclusion

Full repository-managed at-rest encryption is not currently established.

The codebase has implemented data-protection controls and governance mechanisms:

- field-level envelope encryption for selected governed provider credentials
- optional encryption for user/admin-initiated data exports
- storage-layer encryption governance for operator evidence, diagnosis, and audit review
- AEAD JSON persistence for the world-engine JSON store when `RUN_STORE_BACKEND=json_aead`

Those controls do not automatically encrypt every persisted database file, Redis append-only file, local Docker named volume, object-store file, or backup. Production deployments must therefore provide storage-layer encryption and preserve evidence for each persisted surface before the platform can claim full at-rest encryption. The repository now provides the governance and diagnosis path for that evidence, plus an app-managed AEAD option for world-engine JSON persistence when SQL storage is not used.

## Governance and diagnosis

Storage-layer evidence is managed through the existing Security Governance surface:

- Administration Tool: `/manage/security-governance`
- Backend API: `GET/PATCH /api/v1/admin/security/governance`
- Persisted key: `site_settings.security_governance_config`
- Evidence field: `storage_encryption_evidence`
- Diagnosis check: `storage_layer_encryption` in `GET /api/v1/admin/system-diagnosis`

The evidence pack covers these surface ids:

- `backend_sqlite`
- `backend_redis_aof`
- `world_engine_json_run_store`
- `world_engine_sqlalchemy_run_store`
- `langfuse_postgres`
- `langfuse_clickhouse`
- `langfuse_minio`
- `langfuse_redis`
- `backups_snapshots`

Each surface must record `status`, `control_type`, `evidence_ref`, and, for active encryption controls, `key_ref`. Backup and snapshot evidence must also record `restore_test_ref` when restore-test evidence is required.

## Persisted surfaces not fully covered

| Surface | Persistence evidence | Current at-rest status |
| --- | --- | --- |
| Backend SQLite database | `backend/app/config.py` defaults to `sqlite:///backend/instance/wos.db`; `docker-compose.yml` bind-mounts `./backend/instance:/app/instance` | Plain SQLite unless the host filesystem, Docker storage root, or a future SQLCipher/app-managed control encrypts it |
| Backend runtime governance Redis | `docker-compose.yml` runs Redis with `--appendonly yes` and named volume `redis-data:/data` | Plain Redis AOF on the Docker local volume unless the host/volume layer is encrypted |
| World-engine JSON run store | `world-engine/app/config.py` defaults `RUN_STORE_BACKEND=json`; `RUN_STORE_BACKEND=json_aead` writes encrypted `*.json.enc` files | Plain JSON only when the local/default JSON store is used; AEAD-encrypted when `json_aead` and `WORLD_ENGINE_JSON_AEAD_KEY` are configured |
| World-engine SQLAlchemy run store | `world-engine/app/runtime/store.py` stores `payload_json` when `RUN_STORE_BACKEND=sqlalchemy` | Plain application payload unless the selected database/storage layer provides encryption |
| Local Langfuse Postgres, ClickHouse, MinIO, and Redis | `docker-compose.langfuse.yml` declares `langfuse-postgres-data`, `langfuse-clickhouse-data`, `langfuse-clickhouse-logs`, `langfuse-minio-data`, and `langfuse-redis-data` local volumes | No volume encryption is declared by Compose; encryption depends on host, Docker, or managed storage controls |
| Backups and snapshots | `docs/database/README.md` and `docs/operations/README.md` describe backup concepts | No committed backup job proves encrypted backup output or KMS/SSE policy |

## Verification commands

Use these commands to re-check the repository evidence:

```bash
rg -n "SECRETS_KEK|AESGCM|encrypted_secret|encrypted_dek" backend/app backend/migrations .env.example docker-up.py
rg -n "SQLALCHEMY_DATABASE_URI|sqlite:///|backend/instance|RUN_STORE_BACKEND|json_aead|payload_json" backend/app world-engine/app docker-compose.yml
rg -n "appendonly|volumes:|langfuse-.*-data|ENCRYPTION_KEY" docker-compose.yml docker-compose.langfuse.yml .env.example .env.langfuse.example
rg -n "encrypt_export|decrypt_export|AES-256-CBC" backend/app/services backend/docs backend/tests
rg -n "storage_encryption_evidence|storage_layer_encryption|storage_encryption_governance" backend/app administration-tool
```

These commands prove the current control boundaries. They do not prove host-level disk encryption, Docker volume-driver encryption, cloud KMS encryption, or backup encryption; those require deployment evidence outside the repository.

## Completion plan

### 1. Define the production encryption boundary

Choose one supported pattern per deployment:

- Managed encrypted services: PostgreSQL with encrypted storage/KMS, encrypted Redis persistence or Redis without disk persistence, object storage with server-side encryption, encrypted backups/snapshots.
- Self-hosted encrypted storage: full-disk encryption for the host plus encrypted Docker storage root or encrypted volume driver, with documented key custody and boot recovery.
- App-managed SQLite for a small single-node deployment: SQLCipher or equivalent database-level encryption, plus encrypted runtime stores and encrypted backups.

Do not mark the control complete until every persisted surface in the table above is assigned to one of those patterns.

### 2. Close the local SQLite/runtime-store gap

- Keep SQLite as local/dev-only unless the host has documented full-disk encryption.
- If SQLite remains a supported production store, add a SQLCipher-backed configuration path and migration/backup procedure.
- Move world-engine JSON persistence to a SQL-backed store for production, or use `RUN_STORE_BACKEND=json_aead` with `WORLD_ENGINE_JSON_AEAD_KEY` from the production secret store. Rotate the AEAD key through the same key-custody process as `SECRETS_KEK`.
- Ensure backend-local transitional runtime stores follow the same rule if they are enabled outside tests.

### 3. Close Docker volume gaps

- Run Redis AOF, Langfuse Postgres, ClickHouse, MinIO, and Langfuse Redis on encrypted volumes or managed services with encryption enabled.
- Record the storage control per volume: provider/KMS key id, volume driver, host full-disk-encryption state, or managed-service encryption flag.
- Do not treat Compose passwords, `ENCRYPTION_KEY`, or `SECRETS_KEK` as proof of volume encryption.

### 4. Close backup and export gaps

- Require encrypted backup output for SQLite/PostgreSQL, Redis persistence, Langfuse Postgres/ClickHouse, and object storage.
- Store backup encryption keys outside the backup destination.
- Keep encrypted exports available, but label them as export-artifact protection, not database-at-rest protection.
- Plan an export crypto upgrade from CBC-only payloads to an authenticated AEAD format or an authenticated envelope around the current payload format.

### 5. Maintain the operator evidence pack

For each deployment, maintain the Security Governance evidence pack with:

- database encryption setting and KMS/key reference
- Docker/host volume encryption proof, if self-hosted
- Redis persistence encryption or explicit non-persistence decision
- object-storage encryption proof
- backup encryption proof and restore test result
- key-rotation owner and cadence for `SECRETS_KEK`, storage KMS keys, Langfuse keys, and backup keys

The claim "full at-rest encryption is complete" is allowed only when this evidence pack exists and every persisted surface above is covered.
