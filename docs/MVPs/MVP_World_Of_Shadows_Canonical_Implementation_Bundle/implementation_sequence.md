# Implementation Sequence

## Priority Order

1. Establish canonical authority boundaries (authoring -> publish -> runtime -> projection).
2. Stabilize slice-critical runtime behavior and player-visible continuity.
3. Normalize operator/admin diagnostics and release controls.
4. Align API/MCP surfaces with the same authority model.
5. Close remaining drift points and unresolved operational obligations.
6. Run evaluator comparison and decide next investment based on evidence.

## Dependency Chain

- Content authority and publish contracts must be stable before runtime proof claims.
- Runtime proof must exist before broad UX claim expansion.
- Diagnostics and governance instrumentation must be in place before declaring closure.
- Comparative evaluator test must run before platform expansion beyond MVP scope.

## Minimum Execution Waves

### Wave 1: Authority and continuity baseline
- Confirm stage continuity and ownership boundaries.
- Confirm canonical source precedence and activation rules.
- Confirm proposal/validation/commit/render seams are explicitly represented.

### Wave 2: Experience and surface reliability
- Validate player shell obligations.
- Validate admin/operator visibility and incident pathways.
- Ensure consequence carry-forward is perceivable in short sessions.

### Wave 3: Integration and control-plane discipline
- Validate API/MCP boundary behavior.
- Validate governed AI support and retrieval boundaries.
- Keep runtime mode governance explicit (mock/AI/hybrid) where configured.

### Wave 4: Final closure
- Execute no-loss verification.
- Complete mapping-table verification.
- Retire `MVP/` only if deletion gate passes.

## MVP Evaluation Runbook

Run a small controlled comparative evaluation as soon as the slice is stable enough:

- evaluator group: 5-8 participants
- setup: same opening situation for WoS slice and generic baseline
- questions:
  - Is there a clear qualitative difference?
  - Does shaped drama still feel playable?
  - Is validation/commit value visible to players?
