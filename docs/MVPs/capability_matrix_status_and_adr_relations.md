This file is the current Capability Matrix truth map. It is not a wishlist and not a historical idea list. It connects runtime capabilities, stable semantic implementation names, Pi / Π legacy labels, ADR ownership, maturity, runtime contracts, tests, Langfuse / MCP / staging evidence where required, and known promotion blockers.

A capability is only implemented when code, tests, runtime wiring, and required evidence support that claim. Documentation, comments, local trace IDs, test names, PASS labels, or string presence are not enough to promote a row.

### Observability authority (non-negotiable split)

```text
Frontend ≠ observability authority
Backend / world-engine + Langfuse = observability authority
```

```text
Frontend stays visually and functionally largely stable.
Backend / world-engine / Langfuse must supply missing system-level observability.
```

The frontend is **not** the observability authority. The current player shell should remain **visually stable** and continue rendering **backend-provided** player-facing story cards. Dramatic observability must be produced by **backend/world-engine runtime records** and **Langfuse spans/scores**. Frontend changes should be limited to **optional presentation** improvements (e.g. card color tuning, readability, typewriter UX) and **displaying backend-provided diagnostics** — not to inferring dramatic correctness.

The frontend is **not** where beat, narrator, NPC, or capability **logic** should originate. It should keep rendering story cards as today.

The **frontend** may at most **display** what the backend already supplies. It must **not** infer or own answers to:

- Was the beat realized?
- Was the NPC allowed to do that?
- Was the narrator responsible?
- Which capability was selected / violated?

Those decisions belong in the **runtime / backend path**, recorded for **Langfuse** (spans, scores, path summaries), then optionally surfaced in the shell. **BeatRealization UI**, **NarratorAuthority UI**, and **dramatic Capability UI** are **not** frontend obligations unless the backend first emits authoritative fields.

**Division of responsibility:** the **frontend** remains the **play surface**; **Langfuse** is the **analysis surface**; the **world-engine** remains **canonical truth**.

**Frontend status (summary):** Implemented as player-facing rendering shell. Keep visually stable. Optional card color/readability improvements are allowed. Runtime intelligence observability belongs to backend/world-engine and Langfuse, not the frontend.

## ADR-0041 Controlled Runtime Capability Authority

ADR-0041 should be treated as **Controlled Runtime Capability Authority**, not an observability-only sidecar. **Target role:** a central runtime mechanism that classifies the situation, selects semantic capabilities, routes only necessary validators, reduces validation cost, surfaces drift versus legacy seams (`run_validation_seam`), prepares **scoped** authority transfer, and supports controlled runtime co-authority for bounded concerns.

**Current status (implementation):** **Not Finished — local runtime authority readiness implemented; scoped co-authority, readiness preview, enforcement, scoped readiness aggregation pilot, and optional runtime readiness consumer are flag-gated; full commit/readiness authority pending.** Selector, validator execution plan, dispatch (default `dry_run`; opt-in `plan_enforced`), registry inventory, turn-class coverage, LangGraph sidecar local execution, authority bridge, partial-transfer readiness, handoff-candidate policy, `validation_co_authority_decision`, `readiness_co_authority_preview`, `readiness_co_authority_enforcement`/`readiness_policy_input`, and `readiness_aggregation_decision` exist **locally** with bounded, flag-gated behavior. With **`ADR0041_RUNTIME_READINESS_CONSUMER_ENABLED=true`**, the backend player-session bundle may apply aggregation as a **veto-only** overlay on `runtime_session_ready` / `can_execute` (no `validation_outcome` or commit changes). **`run_validation_seam` remains canonical** for `validation_outcome`, commit, and seam/legacy readiness truth **today**.

**Description:** ADR-0041 is the runtime capability authority track. It classifies turn situations, selects semantic capabilities, routes validators, reduces unnecessary checks, compares against legacy seam authority, and prepares scoped transfer of validation authority. Sidecar, projection, preview, and bridge modes are temporary safety phases, not the target end state.

**Current scoped co-authority boundary:** `ADR0041_VALIDATOR_DISPATCH_MODE=plan_enforced` plus `ADR0041_SCOPED_CO_AUTHORITY_ENABLED=true` may emit `runtime_intelligence_projection.validation_co_authority_decision` only when `partial_transfer_ready=true` for the selected turn class. `ADR0041_READINESS_CO_AUTHORITY_PREVIEW_ENABLED=true` may emit policy-grade `readiness_co_authority_preview`; `ADR0041_SCOPED_READINESS_ENFORCEMENT_ENABLED=true` may emit `readiness_co_authority_enforcement` (`allow|block|no_decision`) as a pilot readiness policy input; `ADR0041_SCOPED_READINESS_AGGREGATION_ENABLED=true` (with scoped co-authority, preview, and enforcement all enabled) may emit `readiness_aggregation_decision` for seam-canonical + ADR-0041 veto-only readiness aggregation in projection. **`ADR0041_RUNTIME_READINESS_CONSUMER_ENABLED=true`** may apply that aggregation as a veto-only overlay on the HTTP player-session bundle’s `runtime_session_ready` / `can_execute` (diagnostics: `governance.adr0041_runtime_readiness_consumer`); read-only echoes live under `governance.adr0041_readiness_projection_echo`, Inspector `authority_projection`, and operator turn-history rows. ## Capability Matrix vs Capability Selector

[ADR-0041](../ADR/adr-0041-semantic-capability-selection-and-runtime-capability-budgeting.md) defines the Runtime Capability Authority boundary; semantic capability selection is one stage inside it.

The Capability Matrix answers: what exists, what maturity it has, what ADR governs it, and what evidence is required.

The Capability Selector answers: which semantic capability subset is activated for the current turn.

Rules for future contributors:

- A capability can be implemented but not selected for a given turn.
- A capability can be selected but still fail validation.
- A capability can be observed but not enforced.
- A capability can be enforced locally without being live-verified.
- Selection is not promotion evidence by itself.
- Opening scenes should remain narrator-focused and cost-bounded: typical `enforce` capabilities are `narrator_authority`, `scene_energy`, `environment_state`, `information_disclosure`, and `voice_consistency`; typical `observe` capabilities are `thematic_tracking`, `callback_web`, and `sensory_context`; typical exclusions are `npc_agency`, `player_intent_inference`, `action_resolution`, `consequence_cascade`, `long_horizon_forecast`, `silence_negative_space`, and `dramatic_irony` unless situation signals require them.
- Explicit player turns keep player authority when NPC response evidence is present: `npc_decision_required` or `npc_agency` ledger evidence may select `npc_agency`, but must not reclassify the turn away from `player_input` / `active_actor=player`.

Selector output, manifests, RuntimeAspectLedger evidence, MCP payloads, and Langfuse score names must use semantic names only. Pi / Π labels remain historical index vocabulary and must not become active selector keys or runtime branch keys.

## Trace exports vs static archives

Langfuse / API path summaries may include fields that a **current** `repo` does not yet contain (or the opposite: source ahead of export schema). **Always pair trace timestamp with git SHA.**

Example in **this** repository: `p0_action_resolution_evidence` is built in `[world-engine/app/story_runtime/manager.py](../../world-engine/app/story_runtime/manager.py)` and asserted in `[world-engine/tests/test_story_runtime_api.py](../../world-engine/tests/test_story_runtime_api.py)`. If an archive shows the field in traces but not in source, the archive is stale relative to that trace (or traces were produced from a different branch).

**Current boundary:** Dramatic aspects originate in backend/world-engine runtime records and Langfuse spans/scores. The frontend may render backend-provided metadata, but it must **not** be responsible for *determining* beat realization, narrator/NPC authority, or capability correctness. Runtime aspect records, path summaries, deterministic scores, and MCP extraction now carry first-class aspect fields. Table A rows remain `partially_proven` where fresh staging proof is still environment-dependent; Table B `Π6` is `implemented` for the modular runtime-governance contract itself.

### Observability data flow (target)

**Correct:**

```text
World-Engine decides
→ backend delivers blocks + optional metadata
→ Langfuse stores aspect spans/scores
→ frontend renders as today (optional display of backend fields)
```

**Incorrect:**

```text
frontend interprets visible text
→ frontend decides whether beat/NPC/narrator/capability was correct
```

(Equivalent extended chain: runtime decision → aspect record → validation/commit consumes record → visible output carries origin metadata → Langfuse span + score → optional frontend display.)

Future AI agents and developers reviewing this area should check:

- Matrix claims against source code.
- Source code against tests.
- Tests against actual runtime behavior.
- `RuntimeAspectLedger` fields against documented capabilities.
- Langfuse/MCP scores against semantic capability names.
- ADR references against existing files.
- Local verification logs against reproducible commands, Git SHA/branch, environment, and limitations.
- Partial capabilities against the promotion rules in [capability_matrix_live_claim_gates.md](capability_matrix_live_claim_gates.md).

Do not promote a capability based only on documentation, comments, local trace IDs, test names, PASS labels, or string presence.
