Π5 completion note: The active runtime does not ship a standalone ToT/Reflexion product surface. Its implemented Π5 scope is the bounded live-turn self-correction loop in LangGraph: parser/validation failures plus runtime-aspect authority and dramatic capability failures are converted into feedback-coded rewrites, then exposed through path summaries, Langfuse spans, and score metadata.

Π8 status note: **partial, but no longer diagnostic-tree-only.** The active runtime now has four bounded Π8 surfaces: committed non-opening turns emit the cheap, non-authoritative `branching_forecast.v1`; operator callers can request an isolated bounded multi-turn `branching_simulation_tree.v1`; operator callers can persist that simulation as a durable selectable `branching_tree_record.v1`; and branch-tree lifecycle events are recorded in a bounded durable `branch_timeline_record.v1` with `branch_timeline_event.v1` events and `branch_timeline_snapshot.v1` summaries. Branch-tree records carry root session fingerprints, selectable node ids, stale/expired/committed lifecycle states, and a select-and-commit operation that replays the selected simulated input path through the normal `StoryRuntimeManager` canonical turn path instead of adopting clone state. The timeline tracks create, stale, expire, select, replay-start, replay-commit/replay-conflict, compact, and archive events; it enforces bounded active-tree retention and event compaction for operator evidence. Simulation still uses temporary session clones and remains non-authoritative until selection. Π8 remains partial because preview-package branch simulation, full UI/operator review workflow, and long-horizon or unbounded future-tree execution are still pending. Π9 completion note: The active runtime now ships a standalone, deterministic **context synthesis** layer between retrieval/director state and model routing. It consumes `retrieval`, `context_text`, director scene assessment, semantic/social records, hierarchical memory, validation feedback, and the runtime aspect ledger, then emits a `ContextSynthesisBundle` with evidence items, obligations, conflicts, gaps, lane mix, and bounded diagnostics. On recoverable validation/self-correction passes, validation feedback is resynthesized into a non-authoritative retry prompt block and recorded in `context_synthesis_retry_history`, `validation_feedback`, `resynthesis_count`, and `used_for_self_correction` diagnostics. This is **proposal-support only**: it may shape the model prompt, but it does not decide committed truth, produce final story text, mutate canon, or bypass validation/commit. The implementation is anchored in `[ai_stack/context_synthesis_contracts.py](../../ai_stack/context_synthesis_contracts.py)`, `[ai_stack/context_synthesis_engine.py](../../ai_stack/context_synthesis_engine.py)`, and the `synthesize_context` / retry-resynthesis paths in `[ai_stack/langgraph_runtime_executor.py](../../ai_stack/langgraph_runtime_executor.py)`.

Π10 completion note: The active runtime now has **full deterministic semantic voice classification** for the GoC live path. GoC character voice YAML is compiled into `CharacterVoiceProfileRecord` runtime profiles, including profile dimensions for worldview, register, syntax/rhythm, rhetorical strategy, and phase alignment. The classifier evaluates every spoken line against all active profiles, emits profile rankings, runner-up confidence, dimension-level winners, ambiguity/mixed-signature evidence, and structured findings. The LangGraph validation pass records a `voice_consistency` runtime aspect; policy-declared forbidden language markers can reject an otherwise approved model output, and `strict_rule_engine` mode can reject high-confidence cross-actor semantic voice confusion through `runtime_voice_consistency_v2` with recoverable rewrite feedback. Langfuse runtime-aspect spans/scores and the MCP runtime-aspect matrix expose the same evidence. This is **not** a free-form LLM style judge or copied-prose oracle. Per ADR-0039, `dialogue_examples` remain authoring examples only: they are excluded from serialized runtime profiles and are not used as pass/fail oracles. ### Gate test oracle policy (cross-cutting)

Promotion and validation are meaningless if gate tests **encode accidental output** instead of declared contracts. **[ADR-0039](../ADR/adr-0039-gate-tests-no-hardcoded-oracle-bypass.md)** forbids hardcoded literals as the **primary** oracle for gate and promotion-style regression tests; expectations must trace to schema, OpenAPI, canonical authored content ([ADR-0025](../ADR/adr-0025-canonical-authored-content-model.md)), or reviewable baselines. This tightens how we read **[ADR-0008](../ADR/adr-0008-validation-strategy-explicit-configurable.md)** (strategy-specific tests must still be honest) and **[ADR-0009](../ADR/adr-0009-evaluation-is-a-promotion-gate.md)** (evaluation evidence must not be “string-matched theatre”).

**Operational inventory:** `[docs/governance/gate_oracle_tightness_inventory.md](../governance/gate_oracle_tightness_inventory.md)` tracks gate oracle quality (strict vs brittle), refactor waves, and the latest gate-run evidence. **Staging / live provider credentials:** same document **§10** — use MCP/backend credential retrieval for readiness; never paste or report secrets; `non_mock_generation_pass` and related Langfuse scores stay degraded when credentials are missing. **Waves 1–2** moved prose, payload, route, template, and diagnostics checks onto YAML fixtures, canonical content, AST/import oracles, and subprocess integration proof. **Waves 3–4** centralized contract constants, derived actor expectations from the GoC runtime profile, replaced raw source-substring probes with AST/INI checks, and removed actor-specific integration-test name coupling. **Wave 5** re-audited Π1-Π13, widened the Table B production-control guard, and removed the Quality Lab duplicate aspect-name list. **Wave 6** records Π16 dramatic-irony oracle discipline: selected private-plan provenance, policy-normalized surface modes, prompt-safety absence checks, and ledger/MCP violation fields. Latest full gate evidence before wave 5: `python tests/run_tests.py --suite gates` — **96 passed** on 2026-05-14; latest targeted wave evidence is recorded in §9 of the inventory and in the Π snapshots below.

### Observability authority (non-negotiable split)

```text
Frontend ≠ observability authority
Backend / world-engine + Langfuse = observability authority
```

```text
Frontend stays visually and functionally largely stable.
Backend / world-engine / Langfuse must supply missing system-level observability.
```

The frontend is **not** the observability authority. The current player shell should remain **visually stable** and continue rendering **backend-provided** player-facing story cards. Dramatic observability must be produced by **backend/world-engine runtime records** and **Langfuse spans/scores**. Frontend changes should be limited to **optional presentation** improvements (e.g. card color tuning, readability, typewriter UX) and **displaying backend-provided diagnostics** — not to inferring dramatic correctness.

The frontend is **not** where beat, narrator, NPC, or capability **logic** should originate. It should keep rendering story cards as today.

The **frontend** may at most **display** what the backend already supplies. It must **not** infer or own answers to:

- Was the beat realized?
- Was the NPC allowed to do that?
- Was the narrator responsible?
- Which capability was selected / violated?

Those decisions belong in the **runtime / backend path**, recorded for **Langfuse** (spans, scores, path summaries), then optionally surfaced in the shell. **BeatRealization UI**, **NarratorAuthority UI**, and **dramatic Capability UI** are **not** frontend obligations unless the backend first emits authoritative fields.

**Division of responsibility:** the **frontend** remains the **play surface**; **Langfuse** is the **analysis surface**; the **world-engine** remains **canonical truth**.

**Frontend status (summary):** Implemented as player-facing rendering shell. Keep visually stable. Optional card color/readability improvements are allowed. Runtime intelligence observability belongs to backend/world-engine and Langfuse, not the frontend.

## Trace exports vs static archives

Langfuse / API path summaries may include fields that a **current** `repo` does not yet contain (or the opposite: source ahead of export schema). **Always pair trace timestamp with git SHA.**

Example in **this** repository: `p0_action_resolution_evidence` is built in `[world-engine/app/story_runtime/manager.py](../../world-engine/app/story_runtime/manager.py)` and asserted in `[world-engine/tests/test_story_runtime_api.py](../../world-engine/tests/test_story_runtime_api.py)`. If an archive shows the field in traces but not in source, the archive is stale relative to that trace (or traces were produced from a different branch).

**Current boundary:** Dramatic aspects originate in backend/world-engine runtime records and Langfuse spans/scores. The frontend may render backend-provided metadata, but it must **not** be responsible for *determining* beat realization, narrator/NPC authority, or capability correctness. Runtime aspect records, path summaries, deterministic scores, and MCP extraction now carry first-class aspect fields. Table A rows remain `partially_proven` where fresh staging proof is still environment-dependent; Table B `Π6` is `implemented` for the modular runtime-governance contract itself.

### Observability data flow (target)

**Correct:**

```text
World-Engine decides
→ backend delivers blocks + optional metadata
→ Langfuse stores aspect spans/scores
→ frontend renders as today (optional display of backend fields)
```

**Incorrect:**

```text
frontend interprets visible text
→ frontend decides whether beat/NPC/narrator/capability was correct
```

(Equivalent extended chain: runtime decision → aspect record → validation/commit consumes record → visible output carries origin metadata → Langfuse span + score → optional frontend display.)
