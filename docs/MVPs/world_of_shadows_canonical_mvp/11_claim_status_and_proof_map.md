# Claim status and proof map

## Why this document exists

The canonical MVP now carries more of the real World of Shadows target without pretending that every carried-forward layer is equally live.
That is the right preservation-first move, but it also makes one risk sharper:

- readers can confuse enduring target truth with active proof,
- later consolidations can accidentally promote target-only material to "already live",
- or the opposite can happen and real active-slice claims can get compressed back into vague prose.

This document fixes that by turning the most important claim families into an explicit status map.

## Status reading rule

A claim may be one of five things:

- **Enduring target**: it belongs to the intended World of Shadows system and must not be deleted from the canonical MVP, even if the active repository slice does not prove it equally today.
- **Active slice**: it belongs to the currently strongest proof-bearing vertical slice and therefore matters to current closure judgments.
- **Implemented now**: repository reality supports it directly.
- **Package-proven now**: load-bearing supplied reports or validation bundles support it now, even if this final pass did not rerun every surface directly.
- **Bounded residue**: it remains real but unresolved, partially transitional, or insufficiently replayed for a stronger closure claim.

The same claim can carry more than one status at once.

## Constitutional proof map

The anti-drift laws are more useful when each one has an evidence home.

| Law / seam | Strongest evidence home | Proof expectation |
|---|---|---|
| One truth boundary | canonical runtime laws; service-boundary docs; world-engine ownership | no alternate route, planner, retrieval layer, or operator surface may self-authorize story truth |
| Commit is truth | reference scaffold; narrative-commit tests; GoC turn contracts | visible player truth stays commit-aligned |
| Turn 0 is canonical | runtime maturity packages; scaffold tests; activation path docs | opening birth is real and not just implied |
| Ordinary player route purity | route-governance docs; frontend/backend/player-shell tests | player route remains free of operator-only or preview-only payloads |
| Publish-bound authoritative birth | v17-v18 maturity packages; publish/runtime continuity docs; activation reports | hardened ordinary-player birth uses published artifact identity and provenance |
| Degraded-safe stays explicit | degraded-safety language in complete MVPs and maturity packages | no soft failure may impersonate canonical success |
| Story persistence incidents stay visible | persistence classification docs and closure ledgers | durability gaps must remain classified, not hidden |
| Payloads self-classify | shell/admin/diagnostic surface docs | audience and payload intent remain explicit |
| Release honesty is mandatory | audit packages, residue record, runtime-proof task list | documentation never outruns implementation and proof |

## Promotion rule for future claims

A claim may be promoted upward only when its support changes in reality.

Promotions should follow this order:

1. target-only claim receives implemented repository support,
2. implemented support receives executable direct proof or a strong fresh package proof,
3. package proof receives replayable environment-backed confirmation when the claim matters to closure language,
4. only then may broader runtime-proof wording advance.

## Anti-loss reading rule

Every future consolidation pass should test itself against this document:

- Did any enduring target layer disappear from the canonical family?
- Did any bounded residue get silently promoted to completion?
- Did any active-slice proof claim get blurred into generic product prose?
- Did any new document make retrieval, authoring, diagnostics, transport surfaces, or MCP look like a second truth boundary?
- Did bounded AI-stack / research / improvement support get collapsed back into vague background prose?

If the answer to any of those is yes, the pass is drifting and must be corrected.
