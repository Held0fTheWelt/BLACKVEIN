## W2.2.5.1 Canonical Guard Outcome Model

**Status:** PASS

**What Exists:**
- `GuardOutcome` enum defined in `w2_models.py:213-230` with 4 values:
  - `ACCEPTED` = "accepted"
  - `PARTIALLY_ACCEPTED` = "partially_accepted"
  - `REJECTED` = "rejected"
  - `STRUCTURALLY_INVALID` = "structurally_invalid"
- Enum is string-backed and Enum-typed for explicit serialization
- Comprehensive docstring explaining each category
- All values are semantically distinct and mutually exclusive

**What is Actually Wired:**
- Imported in `turn_executor.py` (line 28) - ✓ used in production path
- Imported in `ai_turn_executor.py` (line 36) - ✓ used in production path
- Not defined elsewhere; no duplicate definitions

**Strongest Evidence:**
- `_compute_guard_outcome()` function (turn_executor.py:379-404) correctly maps all 4 enum values based on accepted/rejected delta counts
- Logic is explicit: if rejected==0 → ACCEPTED, if accepted==0 → REJECTED, if both present → PARTIALLY_ACCEPTED, if both zero → STRUCTURALLY_INVALID
- Every code path returns a GuardOutcome; no null/unknown states

**What is Missing:**
- Expected "canon_blocked" outcome is NOT in enum
- Task.md expected set: `[accepted, partially_accepted, structurally_invalid, canon_blocked]`
- Actual enum: `[accepted, partially_accepted, rejected, structurally_invalid]`
- **Architectural Decision:** canon_blocked is separated from guard_outcome logic; scene-transition blocking is handled by `scene_transition_blocked` event (turn_executor.py:525), not guard_outcome
- This is correct per the design (guard_outcome = "did deltas pass validation", not "did scene change succeed")

**Relevant Files:**
- `backend/app/runtime/w2_models.py:213-230`
- `backend/app/runtime/turn_executor.py:379-404` (uses enum)
- `backend/app/runtime/ai_turn_executor.py:241-251` (maps enum)

## Overall Result

### Overall W2.2.5 Status

**PARTIAL**

#### Rationale

W2.2.5 achieves correct semantic modeling and implementation for the primary production path (success case), but has **asymmetries and test gaps** in error handling and event logging that prevent full gate passage:

1. **Strengths:**
   - GuardOutcome enum is well-designed with 4 clear, distinct values
   - _compute_guard_outcome() logic is sound
   - AIDecisionLog correctly maps all outcomes
   - Success path: TurnExecutionResult, turn_completed event, and AIDecisionLog tell the same story
   - 8 new tests cover success path comprehensively

2. **Critical Gaps:**
   - **Event log asymmetry:** turn_completed includes guard_outcome, turn_failed doesn't
   - **Error path untested:** No test verifying error path guard_outcome handling
   - **Test asymmetry:** Success path fully tested, error path not tested
   - **Cross-surface consistency:** No test verifying all three surfaces (Result, events, AIDecisionLog) tell same story

3. **Hard Constraint Violation (from Task.md line 223):**
   > "Do not mark W2.2.5 ready if TurnExecutionResult, event logs, and AIDecisionLog can still tell different guard stories for the same turn."

   **Violated:** Error path turn_failed event doesn't include guard_outcome, while TurnExecutionResult always has it. These tell different stories to different consumers.

### Gate Decision

**NOT READY TO MOVE TO THE NEXT WAVE**

Conditional on fixes detailed below.

### Top Blockers (Sorted by Severity)

**HIGH:**
1. **Event log asymmetry** - turn_completed includes guard_outcome, turn_failed doesn't
   - Violates hard constraint that all surfaces must tell same story
   - Prevents event-log-only consumers from determining guard_outcome in error cases
   - Repair: Add guard_outcome to turn_failed event payload

2. **Error path test gap** - No test for guard_outcome when execution_status != "success"
   - Cannot guarantee error paths correctly set STRUCTURALLY_INVALID
   - Cannot guarantee exception handling maintains guard_outcome coherence
   - Repair: Add test_guard_outcome_error_path test

3. **Cross-surface consistency untested** - No test verifying all three surfaces agree
   - Cannot prove TurnExecutionResult, event logs, and AIDecisionLog tell consistent story
   - Repair: Add integration test reading all three for same turn

**MEDIUM:**
4. **guard_notes asymmetry** - Present conditionally in success path, always in error path
   - Makes guard_notes presence unreliable as signal
   - Repair: Standardize guard_notes population (either always or conditionally in both paths)

5. **turn_failed payload untested** - No test for error event structure
   - Repair: Add test_turn_failed_event_payload test

---

### False-Completeness Risks

1. **Enum existence vs. wiring:** GuardOutcome enum exists and is imported, but this alone doesn't prove it's actually used. The code IS wired correctly, but future refactors could break this silently without error path tests.

2. **Success path completeness:** The implementation looks complete because success path is well-tested and all surfaces agree. Error path is untested, so asymmetries are not exposed by test suite.

3. **Event log completeness:** turn_completed event includes guard_outcome, creating false impression that "event logs include guard_outcome." turn_failed event contradicts this.

4. **Test quantity vs. coverage:** 8 new tests sounds comprehensive, but only covers success path. Error path, event asymmetries, and cross-surface consistency are untested.

## Conclusion

W2.2.5 **successfully normalizes guard outcome semantics for the success path** and **correctly implements all four canonical outcomes**. However, **error paths expose asymmetries** that violate the hard constraint: "all runtime surfaces must tell the same guard story for the same turn."

**Recommendation:** Implement Phase 1 fixes (2-3 hours) to add guard_outcome to error-path events, then Phase 2 tests (2-3 hours) to verify error path behavior. This will move W2.2.5 to PASS and gate-ready status.

The implementation is **functionally sound** but **incomplete in asymmetry resolution and error-path testing**. It is **not ready for production deployment as-is** because event log consumers cannot reliably determine guard_outcome from error events.
