Runtime authority

Authoritative live play (runs, lobby, WebSocket state, turn execution as exposed by the play service) runs in world-engine/, not inside this Flask process. See docs/architecture/backend_runtime_classification.md.

Backend role

The backend owns accounts, permissions, authored game content, and bootstrap data for the client. It calls the play service over HTTP using PLAY_SERVICE_INTERNAL_URL / PLAY_SERVICE_PUBLIC_URL and shared secrets (PLAY_SERVICE_SHARED_SECRET, optional PLAY_SERVICE_INTERNAL_API_KEY) as implemented in app.services.game_service.

Runs

Create/list/detail/terminate run operations used by the player flow are proxied or mediated through /api/v1/game/* and implemented against the play service HTTP API. Run identifiers and nested-run envelopes follow the contracts validated in game_service and docs/architecture/canonical_runtime_contract.md.

Tickets

Short-lived play tickets are issued by the backend (POST /api/v1/game/tickets) after resolving join context. The browser or frontend uses the ticket and WebSocket base URL from the response to attach to the play service. Ticket TTL is controlled by GAME_TICKET_TTL_SECONDS (clamped in config).

Join context

resolve_join_context maps account, character, display name, and optional preferred role to a participant identity for a given run_id before a ticket is minted. This keeps identity policy on the backend while execution stays on the engine.

Session / bootstrap (non-authoritative)

POST /api/v1/sessions builds an in-memory SessionState for tests, operators, and transitional tooling. API responses explicitly warn that this is not equivalent to the live world-engine runtime. Do not treat it as the source of truth for production play.
