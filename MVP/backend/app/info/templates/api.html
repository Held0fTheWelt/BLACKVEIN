Base URL

All versioned REST endpoints live under /api/v1. There is no bundled OpenAPI/Swagger UI in this service; see docs/api/REFERENCE.md in the repository.

Major areas (implemented)

  • Auth — /api/v1/auth/* (login, refresh, logout, registration flows as exposed by the API).
  • Users & roles — /api/v1/users/*, /api/v1/roles/*.
  • Site & content — news, wiki, slogans, forum, analytics, areas — see REFERENCE.md for exact paths.
  • Game / play bootstrap — /api/v1/game/* (templates, runs, characters, tickets, transcripts) — bridges to the play service where noted in code.
  • Admin JSON APIs — /api/v1/admin/*, /api/v1/game-admin/*, wiki admin, data export — for trusted clients (admin tool or automation), not player HTML.
  • Sessions (non-authoritative) — /api/v1/sessions/* — in-process operator/MCP bridge; responses document that live play is in the world-engine.

Authentication style

Primary API authentication is JWT (Authorization: Bearer <access_token>) via Flask-JWT-Extended. Refresh tokens rotate according to docs/api/REFERENCE.md. Some routes additionally use service tokens (e.g. MCP / automation) as documented per endpoint.

Legacy browser Flask sessions may still exist for compatibility routes under /logout etc.; canonical new clients should use JWT against the API.

CSRF

api_v1 blueprint is CSRF-exempt; browser forms posting to legacy web routes are not the canonical API path. Frontends that use cookies + same-site policies should still follow their own CSRF strategy for any cookie-backed endpoints.

Postman / collections

No Postman collection is shipped inside backend/. If your team maintains one, link it from your internal docs; machine-readable detail remains docs/api/REFERENCE.md.
