Where AI code lives

Rich AI orchestration (e.g. StoryAIAdapter, ai_turn_executor, supervisor_orchestrator, tool loops) lives under app.runtime. These components operate on in-process SessionState for tests, tooling, and the non-authoritative /api/v1/sessions bridge — not as a second live engine.

Proposal vs authority

AI layers produce proposals (structured decisions, narration drafts) that must pass validation / policy in code. For live play, the world-engine is the component that owns committed runtime state and websocket delivery; the backend must not be mistaken for that authority.

Automation webhooks (n8n)

Optional N8N_WEBHOOK_URL / N8N_WEBHOOK_SECRET support signed outbound hooks (e.g. wiki translation triggers in app.n8n_trigger). N8N_SERVICE_TOKEN can gate inbound-style automation to selected admin endpoints. This is operational integration, not game runtime.

Boundary summary

  • Backend — business rules, persistence, ticket issuance, HTTP to play service.
  • World-engine — authoritative run execution and live player connection semantics.
  • In-process AI path — shared libraries and simulation/testing; labeled non-authoritative in architecture docs.
